## 1. Knowledge Makerとは

### 1.1 プロジェクトの目的

Knowledge Makerは、**社内に散らばった業務マニュアル等をAIと人間が読みやすい形で整理し、GitHubのナレッジリポジトリにまとめるための前処理ツール**です。

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                         Knowledge Maker（本ツール）                              │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  📁 input/ に資料を配置                                                         │
│      │  例: 見積書.pdf, マニュアル.xlsx, 手順書.pptx                             │
│      │                                                                          │
│      ↓                                                                          │
│  ┌────────────────────────────────────────────────────────────────────────────┐ │
│  │ 【Step 1】文字起こし（python -m src.km.cli.convert）                        │ │
│  │                                                                            │ │
│  │   🤖 AI-OCR: gpt-5-mini（200並列処理）                                      │ │
│  │                                                                            │ │
│  │   処理内容:                                                                │ │
│  │   ・PDF/Excel/Word/PPTからテキストを抽出                                    │ │
│  │   ・図・表・画像をAI OCRで読み取り、テキスト化                               │ │
│  │                                                                            │ │
│  │   生成されるフォルダ（pre-knowledge/内）:                                   │ │
│  │   ├── 01_input/                  ← 元ファイルのコピー                       │ │
│  │   ├── 02_transcribed_markdown/   ← 文字起こし結果（transcribed.md）        │ │
│  │   └── 04_images/                 ← 抽出した画像ファイル（あれば）           │ │
│  └────────────────────────────────────────────────────────────────────────────┘ │
│      │                                                                          │
│      ↓                                                                          │
│  ┌────────────────────────────────────────────────────────────────────────────┐ │
│  │ 【Step 2】きれい化（python -m src.km.cli.enhance）                          │ │
│  │                                                                            │ │
│  │   🤖 整形AI: gpt-5.1（temp=0, max_tokens=100,000）                          │ │
│  │   📦 大規模ファイル: 3万文字以上は自動分割→3並列処理                        │ │
│  │                                                                            │ │
│  │   処理内容:                                                                │ │
│  │   ・テンプレートに沿ってAIが読みやすく整形                                   │ │
│  │   ・サマリー・目次を自動生成                                                │ │
│  │   ・図の内容を詳細にテキスト化                         　　　　　　　　　　　　 │ │
│  │   ・🔍 品質チェック（文字数・表の行数の2項目を自動検証）                       │ │
│  │   ・1入力ファイル ＝ 1マークダウン                                          │ │
│  │                                                                            │ │
│  │   生成されるフォルダ（pre-knowledge/内）:                                   │ │
│  │   └── 03_formatted_markdown/                                               │ │
│  │       ├── formatted.md           ← きれい化結果                            │ │
│  │       └── quality_report.json    ← 品質チェック結果                        │ │
│  └────────────────────────────────────────────────────────────────────────────┘ │
│      │                                                                          │
│      ↓                                                                          │
│  📁 pre-knowledge/{ドキュメント名}/ に出力完了（作業フォルダ）                    │
│      ├── 01_input/見積書.pdf                                                    │
│      ├── 02_transcribed_markdown/transcribed.md                                 │
│      ├── 03_formatted_markdown/formatted.md  ★きれい化完了                      │
│      └── 04_images/img_001.png, img_002.png ...                                 │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
                                       │
                                       ↓
┌─────────────────────────────────────────────────────────────────────────────────┐
│  【Step 3】ナレッジ化（python -m src.km.cli.deploy）                             │
│                                                                                 │
│   🤖 統合AI: gpt-5.1（要約/構造提案/統合/用語集生成）                             │
│                                                                                 │
│   処理内容:                                                                     │
│   ・複数のformatted.mdを業務単位で統合・再構成                                   │
│   ・フォルダ構成をAIが提案（初回構築時）                                         │
│   ・knowledge/フォルダに整理された形で出力                                       │
│                                                                                 │
│   出力先:                                                                       │
│   └── knowledge/                    ← 最終成果物フォルダ（このリポジトリ内）     │
│       ├── 00_用語集.md, mapping.json, readme.md                                 │
│       ├── 01_業務マニュアル/01_概要.md, 02_見積.md, images/                      │
│       ├── 02_チャット運用/01_基本.md, images/                                   │
│       └── ...                                                                   │
│                                                                                 │
│   ※ 最終的には業務ナレッジリポジトリ（knowledge-SS等）にコピー                   │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 ナレッジメーカーがやること（3ステップ）

| # | 処理 | 説明 | 出力先 |
|---|------|------|--------|
| **1** | **文字起こし（テキスト抽出）** | PDF、Excel、Word、PowerPoint等の資料から中身を抽出。表や図はAI OCRで内容を言語化 | `pre-knowledge/{doc}/02_transcribed_markdown/` |
| **2** | **きれい化** | 抽出したテキストをAIで読みやすいMarkdown形式に整形。**1入力ファイル ＝ 1マークダウン** | `pre-knowledge/{doc}/03_formatted_markdown/` |
| **3** | **ナレッジ化** | 複数のマークダウンを業務単位で統合・再構成し、整理された形で出力 | `knowledge/` |

> 💡 **ポイント**: Step1・2は「1入力ファイル＝1出力ファイル」ですが、Step3で複数ファイルを意味のある単位に統合します

### 1.3 解決する課題

| 課題 | Knowledge Makerによる解決 |
|------|--------------------------|
| 🔍 **業務マニュアルがバラバラで見つからない** | GitHubナレッジリポジトリに一元管理 |
| 👤 **フォーマットがバラバラで読みにくい** | AIと人間が読みやすい統一フォーマットに変換 |
| 📁 **担当者ごとにフォルダ構成が違う** | 業務単位（SS、MDK、MTH、TGK）で整理 |
| ⏰ **問い合わせ対応に時間がかかる** | Cursorチャットボットで即時回答可能に |

### 1.4 対象ユーザー

- **業務担当者**: 業務マニュアルを整理・共有したい方
- **情報システム部門**: ナレッジ管理システムを構築したい方
- **開発者**: ドキュメント変換機能を自分のプロジェクトに組み込みたい方

## 2. クイックスタート

### 2.1 必要な環境

| 項目 | 要件 |
|------|------|
| **Python** | 3.9 以上 |
| **OS** | Windows / macOS / Linux |
| **LibreOffice** | `.doc`, `.xls`, `.ppt` を変換する場合に必要 |
| **Azure OpenAI API Key** | AI OCR、画像説明生成、きれい化を使う場合に必要 |

### 2.2 インストール

```bash
# 1. リポジトリをクローン
git clone https://github.com/your-org/knowledge-maker.git
cd knowledge-maker

# 2. 依存パッケージをインストール
pip install -r requirements.txt
```

**requirements.txt の主要パッケージ:**
```
PyMuPDF>=1.23.0       # PDF処理（ページ単位、画像抽出）
openpyxl>=3.1.0       # Excel(.xlsx)処理
xlrd>=2.0.1           # Excel(.xls)処理
python-docx>=1.0.0    # Word処理
python-pptx>=0.6.0    # PowerPoint処理
pandas>=2.0.0         # CSV処理
Pillow>=10.0.0        # 画像処理
openai>=1.0.0         # AI OCR、画像説明生成
psutil>=5.9.0         # メモリ監視
python-dotenv>=1.0.0  # .env読み込み
```


#### LibreOffice のインストール（.pptx/.doc/.xls 変換に必要）

1. [LibreOffice公式サイト](https://www.libreoffice.org/download/download-libreoffice/) から Windows 版をダウンロード
2. ダウンロードした `.msi` ファイルを実行してインストール
3. `.env` ファイルにパスを設定（デフォルトパスと異なる場合）:

```bash
# .env に追加（パスがデフォルトと異なる場合のみ）
KM_SOFFICE_PATH=C:\Program Files\LibreOffice\program\soffice.exe
```

4. 動作確認:

```powershell
# PowerShellで確認
& "C:\Program Files\LibreOffice\program\soffice.exe" --version
```

> 💡 **ヒント**: LibreOfficeがインストールされていない場合、PDF以外のOfficeファイル（.pptx, .doc, .xls）は変換できません。

### 2.3 環境変数の設定（.envファイル）

プロジェクトルートに `.env` ファイルを作成し、必要な環境変数を設定します。

**最小構成（Azure OpenAI使用時）:**

```bash
# .env
# Azure OpenAI API設定（必須）
AZURE_OPENAI_ENDPOINT=https://your-resource-name.openai.azure.com/
AZURE_OPENAI_API_KEY=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
AZURE_OPENAI_API_VERSION=2024-10-21
AZURE_OPENAI_DEPLOYMENT_VISION=gpt5-mini
AZURE_OPENAI_DEPLOYMENT_CHAT=gpt5.1

# LibreOfficeのパス（デフォルトと異なる場合のみ）
# KM_SOFFICE_PATH=C:\Program Files\LibreOffice\program\soffice.exe
```

**OpenAI API使用時:**

```bash
# .env
OPENAI_API_KEY=sk-xxxxxxxxxxxxx
```

> ⚠️ **セキュリティ**: `.env` ファイルは `.gitignore` に追加し、Gitにコミットしないでください
> 
> 💡 詳細な設定項目は [8. 初期設定](#8-初期設定) を参照してください

### 2.4 基本的な使い方

<a id="step1-文字起こし"></a>

#### 📁 Step 1: 文字起こし（02_transcribed_markdown作成）

`input/` フォルダに処理したい資料を入れて実行：

```bash
# input/フォルダに資料を配置
# 例: input/見積書.pdf, input/マニュアル.xlsx, etc.

# 文字起こし実行
python -m src.km.cli.convert
```

**カスタムフォルダを指定する場合:**

```bash
python -m src.km.cli.convert --input my_documents --output my_pre_knowledge
```

**出力結果:**

```
pre-knowledge/
└── 見積書/
    ├── 01_input/見積書.pdf                  ← 元ファイルのコピー
    ├── 02_transcribed_markdown/transcribed.md ← 文字起こしMarkdown
    └── 04_images/                            ← 抽出画像（あれば）
        └── img_001.png
```

<a id="step2-きれい化"></a>

#### 📝 Step 2: きれい化（03_formatted_markdown作成）

文字起こしMarkdownをテンプレートに沿ってAIで整形：

```bash

# デフォルトテンプレートできれい化実行
python -m src.km.cli.enhance --target pre-knowledge/
```

**テンプレートを指定する場合:**

```bash
# カスタムテンプレートを使用
python -m src.km.cli.enhance --target pre-knowledge/ --template src/km/templates/custom.md
```

**出力結果:**

```
pre-knowledge/
└── 見積書/
    ├── 01_input/
    ├── 02_transcribed_markdown/
    ├── 03_formatted_markdown/formatted.md  ← きれい化Markdown ★
    └── 04_images/
```

<a id="step3-ナレッジ化"></a>

#### 🧠 Step 3: ナレッジ化（knowledge/作成）

##### 🆕 新規作成の場合

`knowledge/` フォルダが空または存在しない場合、AIがフォルダ構成を提案して新規作成します。

```bash
# ナレッジ化実行（新規作成）
python -m src.km.cli.deploy --target pre-knowledge/ --output knowledge/
```

##### 🔄 既存ナレッジを更新する場合

既存のナレッジリポジトリを更新する場合は、**先に `knowledge/` フォルダに既存ナレッジを配置**してから実行します。

```bash
# 1. 既存のナレッジを knowledge/ フォルダに配置
#    例: 別リポジトリからコピー
cp -r ../knowledge-tgk/* knowledge/

# 2. ナレッジ化実行（更新モードで自動実行）
python -m src.km.cli.deploy --target pre-knowledge/ --output knowledge/
```

> 💡 **自動判定**: `knowledge/` フォルダに `mapping.json` や `.md` ファイルがあれば、自動的に**更新モード**になります
> 
> 💡 **AI分析**: 更新モードでは、AIが各ドキュメントを既存ナレッジと照合し、最適なアクション（新規追加/更新/スキップ）を提案します

**出力結果:**

```
knowledge/
├── 00_用語集.md
├── mapping.json                    ← トレーサビリティ
├── readme.md                       ← 全体概要
├── PR/                             ← 更新レポートフォルダ
│   └── UPDATE_REPORT_20250120_103000.md  ← 日付時間入り
├── 01_業務マニュアル/
│   ├── 01_概要・運用ルール.md
│   ├── 02_見積作成フロー.md
│   └── images/
└── 02_チャット運用/
    ├── 01_基本対応.md
    └── images/
```

<a id="コマンドまとめ"></a>

#### 📋 コマンドまとめ

| ステップ | コマンド | 入力 | 出力 |
|---------|---------|------|------|
| **Step1** | `python -m src.km.cli.convert` | `input/` | `pre-knowledge/{doc}/01_input/`, `02_transcribed_markdown/`, `04_images/` |
| **Step2** | `python -m src.km.cli.enhance --target pre-knowledge/` | `pre-knowledge/` | `pre-knowledge/{doc}/03_formatted_markdown/` |

## 3. フォルダ構成

### 4.7 画像抽出・AI説明生成（全形式共通）

全ファイル形式で**統一されたプロンプト**を使用して、画像・図の説明を生成します。

**統一プロンプト:**

```
この図・画像の内容を**完全かつ詳細に**説明してください。
この説明は後でチャットボットが参照するナレッジベースに使用されます。
そのため、図を見なくても内容が完全に理解できるように記述してください。

【必須項目】
1. 図の種類: フローチャート、組織図、タイムライン、グラフ、表、概念図など
2. 主要な要素: すべてのボックス、ラベル、矢印、接続線の内容と関係
3. 数値・データ: 表示されているすべての数字、金額、日付、割合
4. テキスト内容: 図内に書かれているすべてのテキストを省略せず記載
5. 流れ・関係性: 要素間の関係、プロセスの流れ、階層構造
6. 色や強調: 色分けや強調表示がある場合はその意味

【出力形式】
- プレーンテキスト形式で出力
- 文章として自然に読める形式で記述
- 省略せず、すべての情報を網羅
- 日本語で出力
```

<a id="大規模ファイルの自動分割処理"></a>

#### 5.2.1 大規模ファイルの自動分割処理

**なぜ分割が必要？**

AIモデルには一度に処理できる文字数に制限があります。例えば、100ページ超のPowerPointを文字起こしすると **15万文字以上** になることがあり、そのままAIに渡すと：

- ❌ 出力が途中で切れる
- ❌ 情報が抜け落ちる
- ❌ 処理に失敗する

といった問題が発生します。

そこで、**3万文字を超えるファイルは自動的に分割** して処理し、最後に統合することで、情報の欠落なく整形します。

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│              大規模ファイル処理フロー（3万文字以上で自動適用）                    │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  📄 transcribed.md（例: 150,000文字のPowerPoint）                                │
│       │                                                                         │
│       │ ファイルサイズをチェック                                                 │
│       │ → 3万文字以上なら自動的に分割処理へ                                      │
│       │                                                                         │
│  ═════╪═════════════════════════════════════════════════════════════════════════│
│  【Step 1】分割                                                                 │
│  ═════╪═════════════════════════════════════════════════════════════════════════│
│       │                                                                         │
│       ↓ 「## 」見出しを境界として3万文字ごとに分割                               │
│       │ ※見出しがない/少ない場合は文字数で分割                                   │
│       │                                                                         │
│  ┌────┼────┬────────┬────────┬────────┬────────┬────────┐                       │
│  │    ↓    │        │        │        │        │        │                       │
│  │ Chunk1  │ Chunk2 │ Chunk3 │ Chunk4 │ Chunk5 │ Chunk6 │                       │
│  │(30,000) │(30,000)│(30,000)│(30,000)│(30,000)│(30,000)│                       │
│  └─────────┴────────┴────────┴────────┴────────┴────────┘                       │
│       │                                                                         │
│  ═════╪═════════════════════════════════════════════════════════════════════════│
│  【Step 2】並列きれい化（デフォルト3並列 → 高速処理）                             │
│  ═════╪═════════════════════════════════════════════════════════════════════════│
│       │                                                                         │
│       ↓ 各チャンクを同時にAIが整形                                               │
│       │                                                                         │
│  ┌─────────────────────────────────────────────────────────────────────────────┐│
│  │  各チャンクで生成される内容:                                                ││
│  │                                                                             ││
│  │  ┌───────────────────────────────────────────────────────────────────────┐ ││
│  │  │ ## サマリー                                                           │ ││
│  │  │ - このセクションでは〇〇について説明...                               │ ││
```

### 5.3 テンプレートとは

テンプレートは「AIへの整形指示書」です。以下を定義します：

| 定義内容 | 説明 |
|---------|------|
| **出力フォーマット** | セクション構成、見出しレベル、配置順序 |
| **整形ルール** | 表の書き方、画像の配置方法、箇条書きのスタイル |
| **禁止事項** | 情報の省略禁止、文字数削減禁止など |

### 5.5 整形ルール

| カテゴリ | ルール | 説明 |
|---------|--------|------|
| **情報保持** | 省略禁止 | 元ドキュメントの内容をすべて含める。要約・圧縮しない |
| **情報保持** | 文字数維持 | 入力と同等以上の文字数を維持（目安: 100%以上） |
| **見出し** | Slide X禁止 | `## Slide 1`のような無意味な見出しは禁止。内容を反映した名前に変換 |
| **見出し** | 意味のある名前 | 例: `## 1. プロジェクト概要`、`## 2. 提案文章テンプレート` |
| **目次** | グループ化 | 関連する見出しは大カテゴリ（`### 1. 概要と基本フレーム`等）でまとめる |
| **目次** | ページ内リンク | すべての目次項目にページ内リンクを設定（クリックで該当箇所へ移動） |
| **表の処理** | Markdown化 | 表のスクショ画像はMarkdown表に変換 |
| **表の処理** | 複雑な表 | Markdown化が難しい場合は画像を残し、テキストで詳細説明 |
| **図の処理** | 画像保持 | フローチャート等の図は画像を残す |
| **図の処理** | 完全言語化 | 図の内容をチャットボットが回答できるレベルで詳細にテキスト化 |
| **書式** | 引用記法なし | `>` を使用しない |
| **書式** | 箇条書き推奨 | 長い文章は箇条書きに分解して構造化 |
| **パス** | 相対パス | 画像参照は `../04_images/filename.png` 形式 |
| **削除対象** | メタ情報 | ドキュメント情報（抽出日時等）、改訂履歴は削除 |

### 5.6 使用するAIモデル

| 項目 | 値 |
|------|-----|
| **モデル** | `gpt-5.1`（デフォルト） |
| **max_completion_tokens** | `100,000` |
| **temperature** | `0`（決定的出力、要約を防ぐ） |

> 💡 `gpt-5.1`で失敗した場合、自動的に`gpt-5-mini`（max_tokens: 16,384）にフォールバック

### 5.10 品質チェック機能

Step2（きれい化）実行後、**自動的に品質チェックが実行**されます。入力（transcribed.md）と出力（formatted.md）を比較し、情報の欠落がないかを検証します。

#### チェック項目（2項目に絞り込み）

| # | チェック項目 | 条件 | 重要度 |
|---|-------------|------|--------|
| 1 | **文字数チェック** | 85%以下で詳細表示、60%未満でエラー | ⚠️/🔴 |
| 2 | **表の行数チェック** | 80%未満で警告 | ⚠️ |

> 💡 **削除した項目と理由**
> - **画像参照チェック**: 表画像（table_xxx.png）はMarkdown表に変換されるため消える。装飾画像も省略されるため、正常な動作で画像が減る。
> - **数値維持チェック**: 文字数チェックでカバーされている。数値が消えれば文字数も減る。

#### 文字数チェックの詳細

文字数が85%以下になった場合、**健全な減少理由**を分析して表示します。

| 減少理由 | 説明 | 減少量目安 |
|---------|------|-----------|
| 装飾画像省略 | 表紙デザイン、ロゴ、背景パターン等の説明を省略 | 5-15% |
| 文書情報/改訂履歴削除 | 抽出日時、ファイル形式、改訂履歴等を削除 | 1-10% |
| 引用記法削除 | `>` 形式の引用を通常テキストに変換 | 1-3% |
| ラベル削除 | 「図の説明:」等のラベルを削除 | 1-2% |
| 見出し統合 | 複数スライドを1つの見出しにグループ化 | 微量 |

これらの理由で説明できる減少は**情報欠落なし**と判断します。

#### 6.3.3 ファイル生成パターン

各ナレッジファイルは、以下の3パターンのいずれかで生成されます：

| パターン | 条件 | 処理 | AI使用 |
|---------|------|------|--------|
| **分割（split）** | 文字数 ≥ 50,000 かつ 章数 > 1 | 章単位で分割してコピー | なし |
| **そのまま（single）** | 文字数 < 50,000 かつ 単一ソース | 全章をコピー | なし |
| **マージ（merge）** | 複数ソースを統合 | AIでリライト統合 | あり |

```
【分割】業務設計資料.pptx (43章)  ─┬→ 01_概要.md (章1-15)
                                   ├→ 02_見積.md (章16-30)
                                   └→ 03_契約.md (章31-43)

【そのまま】FAQ_申込.xlsx (5章)   ──→ 04_FAQ.md (全章)

【マージ】第1章.pdf + FAQ.xlsx   ──→ 05_申込フロー.md (AIリライト)
```

### 6.6 フォルダ構成提案の設計ルール（参考）

AIがフォルダ構成を提案する際の設計ルールです。

**提案のルール:**
- **無理に統合しない**: 関連性が明確なもののみ統合
- **1ファイル = 1ナレッジも正しい選択肢**: 迷う場合は統合しない
- **統合の判断基準**: 同じ業務フロー、同じトピック、補足関係にあるもののみ

**分割ルール:**

| 章数 | 処理 |
|------|------|
| 10章以下 | 関連性があればマージを検討 |
| 10-20章 | 理想的なサイズ（1ファイルに収める） |
| 30章以上 | 必ず分割（1グループ10-20章が目安） |

**命名ルール:**
- **カテゴリ名**: 内容を反映した名前（例: `01_業務マニュアル/`, `02_チャット対応/`）
- **ファイル名**: 内容ベースの名前（日付・記号は自動削除）
  - Before: `01_★250708_チャットのやり取りの王道.md`
  - After: `01_チャット対応全体像と見積提示フロー.md`

**出力順の考え方:**
- **カテゴリ（フォルダ）の順番**: 上から読んでいくと理解しやすい順に並べる（業務の基礎・全体像 → 個別業務 → 応用・例外 → 参考資料）
- **各カテゴリ内のmdファイルの順番**: 理解しやすい順に並べる（業務フローなら: 問合せ受付 → 見積作成 → 見積提示 → 受注 → 施工依頼 → 施工完了）
- 章番号の順番ではなく、**内容の論理的な順番**で配置

### 10.1 PyMuPDF (fitz) - PDF処理ライブラリ

**PyMuPDF (fitz)** は、MuPDFライブラリのPythonバインディングです。
高速なPDF処理と豊富な機能を備えており、Knowledge MakerではPDF・Word・PowerPointの文字起こしに使用しています。

#### pdfplumber との比較

| 項目 | PyMuPDF (fitz) | pdfplumber |
|------|----------------|------------|
| **速度** | ⚡ 非常に高速 | 🐢 やや遅い |
| **テキスト抽出** | ✅ 座標付きで取得可能 | ✅ 座標付きで取得可能 |
| **表検出** | ✅ `find_tables()` で検出可能 | ✅ 表検出に強い（主な用途） |
| **埋め込み画像抽出** | ✅ `get_images()` で直接抽出 | ❌ 非対応 |
| **描画オブジェクト** | ✅ `get_drawings()` で線・図形を取得 | ❌ 非対応 |
| **ページレンダリング** | ✅ ページ→画像変換（スクショ）可能 | ❌ 非対応 |

#### PyMuPDF を選んだ理由

1. **埋め込み画像を直接抽出できる** - 写真・ロゴ・グラフなどを個別に取り出せる
2. **描画オブジェクトを検出できる** - フローチャートや組織図などのベクター図形を画像化できる
3. **ページをレンダリングできる** - 表部分をスクリーンショットして画像化できる
4. **高速** - 大量のPDFを処理する際にも実用的な速度

#### 主要な機能（コードでの使用例）

```python
import fitz  # PyMuPDF

doc = fitz.open("input.pdf")
page = doc[0]

# テキスト抽出（座標情報付き）
text_dict = page.get_text("dict")

# 埋め込み画像の取得
images = page.get_images(full=True)

# 表の検出とスクリーンショット
tables = page.find_tables()

# 描画オブジェクトの取得（フローチャート等の検出に使用）
drawings = page.get_drawings()

# ページの一部を画像化（表・図のスクリーンショット）
pix = page.get_pixmap(clip=rect)
```

### 10.2 openpyxl - Excel処理ライブラリ

**openpyxl** は、Excel 2010以降の `.xlsx` ファイルを読み書きするためのPythonライブラリです。

#### data_only オプション

```python
from openpyxl import load_workbook

wb = load_workbook(filename="input.xlsx", data_only=True)
```

| オプション | 取得される値 |
|-----------|-------------|
| `data_only=False`（デフォルト） | 数式そのまま（例: `=SUM(A1:A10)`） |
| `data_only=True` | **計算結果**（例: `12345`） |

Knowledge Makerでは **`data_only=True` を明示的に指定**しています。
これにより、数式ではなく**実際の値**を取得してMarkdown化します。

#### Markdown表生成の仕組み

ExcelのセルからMarkdown表への変換は、**AIを使わずPythonコードで直接生成**しています：

```python
def _extract_sheet_text(self, worksheet, max_rows: int) -> str:
    # 各セルの値を取得
    for row in worksheet.iter_rows(min_row=1, max_row=rows_to_process):
        formatted = [_format_cell(cell.value) for cell in row]
        rows.append(formatted)
    
    # Markdown表形式に変換
    header_line = "| " + " | ".join(header) + " |"       # ヘッダー行
    separator_line = "| " + " | ".join(["---"] * max_cols) + " |"  # 区切り行
    data_lines = ["| " + " | ".join(row) + " |" for row in data_rows]  # データ行
    
    return "\n".join([header_line, separator_line, *data_lines])
```

#### セル値の整形ルール

| 処理 | 説明 |
|------|------|
| 改行 → `<br>` | セル内の改行を `<br>` に変換（Markdown表で改行表示） |
| パイプ → `\|` | `\|` をエスケープ（表が壊れるのを防ぐ） |
| 空セル → 空文字 | `None` は空文字に変換 |
| bool → TEXT | `TRUE`/`FALSE` 文字列に変換 |

> 💡 AIが関与するのは、**埋め込みグラフや図形の説明生成（Vision API）のみ**です。
> セルデータの抽出・表変換は100%コードで行っています。
